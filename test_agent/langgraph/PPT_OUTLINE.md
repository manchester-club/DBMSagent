# PPT 汇报大纲：基于 LangGraph 的智能测试 Agent 系统

## 第一部分：项目背景与目标

### Slide 1: 封面
- 标题：基于 LangGraph 的智能测试 Agent 系统
- 副标题：多 Agent 协作的代码覆盖率测试与单元测试生成
- 汇报人、日期

### Slide 2: 项目背景
- **问题挑战**：
  - 大型代码库（PostgreSQL）的覆盖率测试复杂
  - 需要理解代码结构、调用关系、覆盖率数据
  - 需要生成和执行 SQL 测试用例
  - 传统方法效率低、难以自动化

### Slide 3: 项目目标
- **核心目标**：
  - 构建智能化的多 Agent 测试系统
  - 自动化代码覆盖率分析和改进
  - 支持多种测试类型（覆盖率测试、单元测试）
  - 提供可扩展的架构设计

## 第二部分：已完成工作

### Slide 4: 技术选型
- **LangGraph**：状态图框架，支持复杂 Agent 编排
- **LangChain**：LLM 集成和工具调用
- **Ollama**：本地 LLM（Qwen3:32b）
- **PostgreSQL**：目标测试代码库

### Slide 5: 核心工具实现（5个工具）
- **Get_Code_Context**：代码上下文查询
- **Traverse_Call_Graph**：调用图遍历
- **Collect_Coverage**：覆盖率收集与分析
- **Search_Nearest_Seed**：SQL 种子查找
- **Run_SQL_Test**：SQL 测试执行

### Slide 6: Coverage Multi-Agent 架构（已实现）
- **架构图**：Supervisor + 3个专家 Agent
- **Supervisor**：智能路由决策
- **Code Explorer**：代码探索
- **SQL Tester**：SQL 执行
- **Coverage Analyzer**：覆盖率分析与改进

### Slide 7: 关键特性 - 智能路由
- **程序化强制规则**：确保关键步骤不遗漏
- **LLM 路由决策**：基于对话上下文智能路由
- **多轮工具调用**：支持复杂的多步骤任务

### Slide 8: 关键特性 - SQL 节点动态创建机制
- **问题发现**：relations 文件中的 SQL ID 引用对应的节点未创建
- **解决方案**：通过 `Collect_Coverage` 和 `Run_SQL_Test` 动态创建 SQL 节点
- **数据流**：relations → 工具执行 → SQL 节点创建 → Search_Nearest_Seed 查找

### Slide 9: 关键特性 - 程序化强制机制
- **问题**：LLM 可能生成 SQL 但忘记执行
- **解决方案**：检测 SQL 生成但未执行的情况，自动注入提示强制执行
- **效果**：确保 SQL 一定会被执行，从而创建 SQL 节点

### Slide 10: 关键特性 - 结构化测试报告
- **Summary 节点**：汇总所有 Agent 的执行结果
- **报告结构**：
  - 测试目标
  - 源代码分析
  - 调用关系分析
  - 覆盖率分析
  - SQL 种子查找
  - 覆盖率改进方案

### Slide 11: 代码质量与架构
- **模块化设计**：`nodes/` 目录，每个工具独立文件
- **工具化封装**：使用 `@tool` 装饰器
- **缓存机制**：节点和边数据缓存，提升性能
- **错误处理**：完善的异常处理和错误提示

### Slide 12: 测试与验证
- **测试用例**：EncodeSpecialDate 函数测试
- **功能验证**：
  - ✅ 代码探索功能正常

## 第三部分：架构设计（未来规划）

### Slide 13: 两层路由架构设计
- **设计理念**：
  - 顶层 Supervisor：决定使用哪个测试子图
  - 子图路由器：子图内部的 Agent 调度
  - 完全独立的子图：互不干扰
  - 共享 Agent：Code Explorer 可被多个子图使用

### Slide 14: 两层路由架构图
- **架构图展示**：
  - Top Supervisor
  - Coverage Subgraph（覆盖率测试子图）
  - Unit Test Subgraph（单元测试子图）
  - 各子图内部的 Agent
  - Summary 节点

### Slide 15: 架构优势
- **职责分离**：顶层决定策略，子图负责执行
- **完全独立**：不同测试类型互不干扰
- **易于扩展**：添加新子图和新 Agent 简单
- **资源共享**：共享 Agent 提高效率
- **统一协调**：通过顶层 Supervisor 统一管理

## 可能的提问与回答

### Q1: 为什么选择 LangGraph？
**A**: LangGraph 提供了状态图框架，非常适合多 Agent 协作场景，支持复杂的工作流编排和状态管理。

### Q2: SQL 节点为什么需要动态创建？
**A**: relations 文件中的 SQL ID 是预先存在的引用，但对应的 SQL 节点需要通过实际执行 SQL 测试来创建，这确保了数据的准确性和一致性。

### Q3: 两层路由架构的优势是什么？
**A**: 职责分离清晰，顶层决定策略，子图负责执行；完全独立的子图互不干扰；易于扩展新的测试类型。

### Q4: 如何保证 LLM 的可靠性？
**A**: 我们采用了程序化强制机制，检测关键步骤是否执行，如果 LLM 遗漏了关键步骤，系统会自动注入提示强制执行。

### Q5: 性能如何？
**A**: 目前实现了缓存机制，未来计划支持并行执行和更智能的缓存策略。
